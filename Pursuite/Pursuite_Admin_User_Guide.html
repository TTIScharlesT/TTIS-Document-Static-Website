Document Purpose

This guide explains how to administer Pursuite in a professional, controlled, and audit-ready manner. It is written for users who have responsibility for the tenant itself rather than only one pursuit, proposal, opportunity, or workspace.

Use this guide to:

  • Configure a new Pursuite tenant.
  • Manage users, roles, workspace access, and offboarding.
  • Configure organization-level security controls.
  • Administer SSO, MFA, AI providers, integrations, and usage guardrails.
  • Govern RFP Shredder, CUI handling, templates, content, reports, and audit evidence.
  • Operate Pursuite in a way that supports proposal quality, compliance, and repeatable business processes.

This guide does not replace company policy. Where your internal cybersecurity, legal, contracts, records-management, or export-control policies are stricter than this guide, follow your internal policy.

Administrator Operating Model

Pursuite administrators own the operating environment. They do not need to perform every proposal action, but they are accountable for ensuring that the platform is configured correctly and that the right users can perform the right work at the right time.

Administrative Principles

Use these principles when making tenant changes:

  1. Least privilege. Give users the minimum role required for their responsibilities.
  2. Separation of duties. Avoid concentrating proposal approval, security administration, billing, and audit export responsibilities in one person.
  3. Traceability. Prefer actions that preserve audit history over destructive cleanup.
  4. Repeatability. Use templates, playbooks, and scheduled review cycles instead of ad hoc administration.
  5. Controlled automation. Enable AI and integrations only after reviewing data-handling and approval requirements.
  6. Business continuity. Maintain more than one active Admin account and document emergency access procedures.

Administrative Roles and Access Boundaries

Pursuite uses system-level roles and proposal workspace roles. System roles govern broad platform capabilities. Workspace roles govern what a user can do inside a specific proposal.

System Roles

Role Purpose Typical Capabilities Administrative Notes
Admin Full tenant administrator User management, tenant settings, security settings, integrations, audit exports, billing, broad access to business records Use sparingly; review quarterly
Proposal Manager Capture/proposal lead Create and manage opportunities and proposals, manage workspace teams, upload and confirm RFP requirements, run proposal workflows Appropriate for proposal leads and capture managers
Writer Contributor Draft assigned sections, view proposal context, use permitted AI drafting tools Should not administer RFP requirement sets or tenant settings
Reviewer Review participant Review content, comment, participate in review workflows Often used for SMEs, color team reviewers, and quality reviewers
Executive Approver Leadership approver Bid/no-bid approval and executive-level review Should be limited to users with decision authority
Read Only Observer/auditor View permitted records without editing Useful for auditors, temporary observers, and management visibility

Workspace Roles

Workspace roles are scoped to an individual proposal. A user may be a Writer on one proposal, a Reviewer on another, and have no access to a third.

Typical workspace administration tasks:

  1. Add team members to a proposal.
  2. Assign the appropriate workspace role.
  3. Assign section ownership.
  4. Review membership before major gates such as color reviews or final export.
  5. Remove users when they no longer need access.

Access Review Standard

Perform an access review at least monthly and before each major proposal deadline. Confirm:

  • Admin list is still accurate.
  • Proposal Managers still need manager-level access.
  • Writers and Reviewers are assigned only to active work.
  • Former employees or subcontractors are deactivated.
  • Read-only access is still justified.
  • SSO group mappings still reflect approved role assignments.

Initial Tenant Implementation

This section provides a complete step-by-step setup procedure for a new tenant.

Phase 1: Establish Administrative Control

  1. Sign in with the initial Admin account.
  2. Confirm the tenant name and organization identifier.
  3. Add at least one backup Admin.
  4. Confirm both Admins can sign in.
  5. Record the Admin owners in your internal system inventory or operations runbook.

Success criteria:

  • At least two Admins exist.
  • Admin accounts are associated with named employees, not shared mailboxes.
  • Emergency access handling is documented.

Phase 2: Configure Organization Defaults

  1. Open organization settings.
  2. Confirm organization display name.
  3. Confirm primary domain.
  4. Set default timezone.
  5. Confirm currency, date, and number formatting where available.
  6. Review enabled feature flags and plan entitlements.
  7. Save changes.

Success criteria:

  • Tenant settings match company identity.
  • Timezone aligns with report schedules and proposal operations.
  • Enabled features match purchased plan and approved deployment model.

Phase 3: Configure Identity and Security

  1. Decide whether SSO is required.
  2. Configure SSO before inviting most users.
  3. Test SSO with one pilot account.
  4. Configure MFA policy.
  5. Confirm Admins and Proposal Managers are enrolled in MFA.
  6. Document support steps for MFA reset and locked accounts.

Success criteria:

  • SSO sign-in works for a test user.
  • MFA is enforced according to company policy.
  • A break-glass procedure exists if SSO becomes unavailable.

Phase 4: Configure AI and Data Controls

  1. Identify approved AI provider or providers.
  2. Add provider credentials.
  3. Test provider connectivity.
  4. Confirm approved use cases.
  5. Confirm CUI/ITAR data handling policy.
  6. Configure CUI external-processing approval only if formally approved.
  7. Review audit logging for AI actions.

Success criteria:

  • AI provider test succeeds.
  • AI data-handling policy is documented.
  • CUI processing state matches organizational approval.

Phase 5: Invite Users and Establish Governance

  1. Create or invite Admins, Proposal Managers, Writers, Reviewers, and Executive Approvers.
  2. Verify users can sign in.
  3. Publish baseline templates.
  4. Approve core content library items.
  5. Configure integrations.
  6. Create scheduled reports.
  7. Review audit export procedure.

Success criteria:

  • Users have correct roles.
  • Proposal Managers can create workspaces.
  • Core templates and content are available.
  • Integrations are tested before operational use.

Organization Profile and Tenant Settings

Organization settings affect how Pursuite behaves across the tenant.

Step-by-Step: Review Organization Settings

  1. Open the Admin or Settings area.
  2. Select Organization or Tenant Settings.
  3. Review the organization name.
  4. Review default timezone.
  5. Review plan and enabled modules.
  6. Review notification defaults.
  7. Review any branding or export defaults.
  8. Save only if changes are required.
  9. Record major setting changes in your internal change log.

Professional Controls

Maintain the following controls:

  • Review tenant settings after plan changes.
  • Review timezone before creating scheduled executive reports.
  • Confirm display name before sharing exports externally.
  • Review feature availability before announcing new workflows to users.
  • Avoid changing organization identifiers during active proposal deadlines unless necessary.

User Lifecycle Management

User lifecycle administration covers onboarding, role changes, temporary access, deactivation, and audit review.

Step-by-Step: Invite a User

  1. Open the user administration page.
  2. Click Add User or Invite User.
  3. Enter the user's full name and email address.
  4. Select the appropriate system role.
  5. Add optional notes if your organization tracks request context.
  6. Send the invitation.
  7. Confirm the user receives the invitation.
  8. Confirm the user signs in successfully.
  9. Add the user to proposal workspaces only where needed.

Recommended evidence:

  • Access request ticket.
  • Approver name.
  • Assigned role.
  • Date access was granted.

Step-by-Step: Change a User's System Role

  1. Open the user list.
  2. Search for the user.
  3. Open the user profile.
  4. Review current system role.
  5. Confirm the requested new role has a business justification.
  6. Change the role.
  7. Save.
  8. Review proposal workspace memberships separately.
  9. Notify the user and requestor.

Controls:

  • Admin role changes should require explicit approval.
  • Proposal Manager role changes should be reviewed for active proposal impact.
  • Role elevation should be time-bound when access is temporary.

Step-by-Step: Deactivate a User

  1. Confirm the offboarding request.
  2. Identify active proposal assignments.
  3. Reassign sections, reviews, reports, templates, and scheduled exports.
  4. Remove the user from active proposal workspaces.
  5. Deactivate the user account.
  6. Confirm the user can no longer sign in.
  7. Review recent audit activity if required.
  8. Retain audit history.

Do not delete audit records for a deactivated user. Deactivation preserves traceability while preventing new access.

Step-by-Step: Review User Access

  1. Export or open the current user list.
  2. Filter active users.
  3. Review Admins first.
  4. Review Proposal Managers.
  5. Review users with CUI or export-controlled access.
  6. Review inactive project participants.
  7. Confirm workspace membership for active proposals.
  8. Remove unnecessary access.
  9. Record review completion.

Role and Permission Governance

Role governance ensures users have appropriate access for their duties.

Role Assignment Guidance

Need Recommended Role
Manage tenant, integrations, billing, security, audit exports Admin
Lead capture/proposal work and manage proposal team Proposal Manager
Draft proposal sections Writer
Review proposal content without editing tenant settings Reviewer
Approve bid/no-bid decisions Executive Approver
Observe work or support audit review Read Only

Common Anti-Patterns

Avoid these patterns:

  • Making every senior user an Admin.
  • Keeping former subcontractors active after proposal closeout.
  • Granting Proposal Manager access only to allow one-time edits.
  • Using shared accounts for proposal or admin work.
  • Letting SSO groups drift from approved role assignments.

Emergency Access

If emergency Admin access is needed:

  1. Document the reason.
  2. Obtain approval according to company policy.
  3. Grant the minimum required role.
  4. Perform the required action.
  5. Remove elevated access when complete.
  6. Review audit logs after the event.

Proposal Workspace Administration

Admins usually do not manage every proposal day to day, but they support governance and recovery when workspace access or gates are blocked.

Step-by-Step: Add a User to a Proposal Workspace

  1. Open the proposal workspace.
  2. Open the Team panel.
  3. Click Add Member.
  4. Select the user.
  5. Select the workspace role.
  6. Save.
  7. Assign sections if applicable.
  8. Confirm the user can access the workspace.

Step-by-Step: Correct a Workspace Role

  1. Open the proposal workspace.
  2. Open the Team panel.
  3. Locate the user.
  4. Review current workspace role.
  5. Select the corrected role.
  6. Save.
  7. Confirm the user can perform the required action.

Proposal Gate Governance

Admins should understand the major gates:

Gate What It Controls Who Usually Acts
Bid/no-bid approval Whether pursuit should proceed Executive Approver
RFP requirement confirmation Unlocks Compliance Matrix and Autonomous Pipeline Proposal Manager or Admin
Low-confidence RFP review Prevents uncertain requirements from being confirmed without human review Proposal Manager or Admin
Pipeline human review gates Controls AI pipeline advancement Assigned reviewer/approver
Final export readiness Ensures content is approved before delivery Proposal Manager

Admin Intervention Rules

Admins should intervene only when:

  • The assigned Proposal Manager is unavailable.
  • Access is misconfigured.
  • A compliance or security concern requires action.
  • A production workflow is blocked by administrative settings.

Record the reason for significant interventions.

Security Administration

Security administration includes user controls, authentication, data classification, audit monitoring, and incident response.

Security Baseline

Recommended baseline:

  • MFA enforced for Admins and Proposal Managers.
  • SSO enabled for enterprise tenants.
  • At least two Admins, no shared Admin accounts.
  • Quarterly access review.
  • Immediate deactivation for offboarded users.
  • CUI/ITAR controls enabled when controlled data is stored.
  • AI provider approval documented.
  • Audit exports reviewed and retained according to policy.

Step-by-Step: Perform a Security Review

  1. Review active Admin accounts.
  2. Review MFA enrollment.
  3. Review SSO status and recent sign-in issues.
  4. Review users added since the last review.
  5. Review role elevations.
  6. Review CUI classification events.
  7. Review failed access attempts or unusual activity.
  8. Review integrations and active credentials.
  9. Document findings and remediation actions.

Incident Response Support

If suspicious activity is reported:

  1. Preserve audit records.
  2. Identify affected users, proposals, and documents.
  3. Disable or restrict affected accounts if required.
  4. Review recent audit events.
  5. Review integrations and tokens.
  6. Escalate according to company incident response procedures.
  7. Document containment and remediation.

SSO and MFA Administration

SSO Configuration Procedure

  1. Confirm the identity provider owner.
  2. Confirm required metadata, redirect URLs, and certificate details.
  3. Configure Pursuite SSO settings.
  4. Configure the application in the identity provider.
  5. Test with one pilot user.
  6. Confirm role mapping behavior.
  7. Test sign-out and re-authentication.
  8. Enable SSO for the broader tenant.
  9. Monitor sign-in errors after rollout.

MFA Configuration Procedure

  1. Open security settings.
  2. Review current MFA policy.
  3. Choose the enforcement level.
  4. Communicate the rollout date.
  5. Require enrollment for Admins first.
  6. Require enrollment for Proposal Managers.
  7. Expand to all users if required.
  8. Monitor support requests.

MFA Reset Procedure

  1. Verify the user's identity through approved internal process.
  2. Confirm the account email.
  3. Reset MFA enrollment.
  4. Ask the user to enroll a new factor.
  5. Confirm successful sign-in.
  6. Record the reset if your policy requires it.

CUI, ITAR, and Data Classification Administration

Pursuite supports classification workflows for controlled information. Admins are responsible for ensuring those controls match organizational policy.

Classification Concepts

Concept Meaning
INTERNAL Default non-CUI business data classification
CUI Controlled Unclassified Information requiring additional handling
ITAR/EAR Export-controlled data requiring strict access and dissemination controls
CUI category Specific CUI category such as controlled, specified, ITAR/EAR, or FOUO legacy
Minimum clearance User clearance threshold required for access when configured

Step-by-Step: Classify an RFP Document as CUI

  1. Confirm the document contains CUI or controlled content.
  2. Confirm the classifier is authorized.
  3. Open the classification controls for the RFP document.
  4. Select the appropriate CUI category.
  5. Enter designator, caveats, reason, and review date when required.
  6. Save classification.
  7. Confirm the document classification shows as CUI.
  8. Confirm access restrictions and audit logging behavior.

External AI Processing Gate

When an RFP document is classified as CUI, the RFP Shredder worker checks whether the organization is approved for external CUI processing before sending document text to an AI provider.

Do not enable external CUI processing unless:

  • The AI provider is approved for the data type.
  • Contract terms permit that processing.
  • Security and compliance owners have approved the configuration.
  • The approval is documented.

Step-by-Step: Remove CUI Classification

  1. Confirm the document no longer requires CUI handling.
  2. Confirm approval to downgrade.
  3. Open the classification record.
  4. Remove or downgrade the classification to none.
  5. Save.
  6. Confirm the RFP document returns to INTERNAL classification.
  7. Retain the classification audit history.

AI Provider Administration

AI provider settings power drafting, RFP Shredder, Compliance Matrix, Autonomous Pipeline, Ask, analysis, and related AI features.

Step-by-Step: Add or Update an AI Provider

  1. Open AI Provider settings.
  2. Select provider type.
  3. Enter endpoint, deployment, model, and API key details as required.
  4. Save the provider.
  5. Run the connectivity test.
  6. Confirm a successful response.
  7. Configure default routing if multiple providers exist.
  8. Record approval and key owner.

Provider Governance Checklist

  • Provider is approved by security/legal.
  • Provider supports required data handling.
  • Key owner is documented.
  • Key rotation schedule is defined.
  • Test result is successful.
  • Usage monitoring is enabled where available.
  • CUI processing approval matches classification policy.

Step-by-Step: Rotate Provider Credentials

  1. Generate a new key in the provider portal.
  2. Add or update the key in Pursuite.
  3. Test connectivity.
  4. Confirm AI workflows run normally.
  5. Revoke the old key.
  6. Record the rotation date.

RFP Shredder Administration

The RFP Shredder converts solicitation documents into structured requirements. Admins should understand both the user workflow and the controls around it.

Supported Files

  • PDF
  • DOCX
  • Maximum file size: 50 MB

PDF files use text extraction with OCR fallback for scanned or image-only PDFs. DOCX files use Word document text extraction and do not use OCR.

Manager/Admin Actions

Proposal Managers and Admins can:

  • Upload RFP files.
  • Replace RFP files.
  • Retry processing.
  • Re-process completed documents.
  • Add manual requirements.
  • Edit extracted requirements.
  • Delete false positives.
  • Mark low-confidence rows reviewed.
  • Confirm requirements.

Writers can view completed requirements, but they cannot perform write or confirmation actions.

Step-by-Step: Resolve a Blocked RFP Confirmation

  1. Open the proposal workspace.
  2. Open RFP Shredder.
  3. Confirm document status is completed.
  4. Look for rows marked Review required.
  5. For each flagged row, compare it to the source solicitation.
  6. Edit the row if the text or classification is wrong.
  7. Mark the row reviewed if it is correct.
  8. Confirm requirements.
  9. Verify Compliance Matrix and Autonomous Pipeline unlock.

Re-Process Behavior

Re-processing follows replace-on-success semantics:

  1. Existing visible requirements remain available while the new run is processing.
  2. If the new run succeeds, old active rows are soft-deleted and replaced with the new extracted set.
  3. If the new run fails, old visible rows remain intact.
  4. Soft-deleted rows remain available for history and audit.

This behavior protects proposal teams from losing a usable requirement set because of a failed re-processing attempt.

Compliance and Audit Administration

Audit and compliance administration supports SOC 2, CMMC, internal control reviews, and customer-driven evidence requests.

Audit Event Categories

Common audited actions include:

  • User sign-in and access-sensitive events.
  • User and role administration.
  • Proposal and opportunity changes.
  • RFP upload, shredding, confirmation, and requirement edits.
  • CUI classification and CUI access.
  • AI provider usage and AI workflow events.
  • Integration and export activity.

Step-by-Step: Export Audit Evidence

  1. Open the audit or compliance area.
  2. Select the date range.
  3. Select action categories if available.
  4. Filter by user, target, or module if needed.
  5. Generate the export.
  6. Confirm the export opens correctly.
  7. Save it to the approved evidence repository.
  8. Record the export purpose and requestor.

Evidence Handling Rules

  • Export only the scope needed.
  • Do not edit raw evidence files.
  • Store exports in approved locations.
  • Restrict access to evidence containing CUI or sensitive proposal data.
  • Retain exports according to company policy.

Templates and Content Library Governance

Templates and approved content are central to repeatable proposal operations.

Template Governance

Admin responsibilities:

  • Publish standard templates.
  • Approve or reject publish requests.
  • Archive obsolete templates.
  • Maintain bid-type and section metadata.
  • Review template usage and quality.

Step-by-step publish review:

  1. Open Templates.
  2. Filter to pending review if applicable.
  3. Open the requested template.
  4. Review title, bid type, volumes, sections, instructions, and metadata.
  5. Confirm it follows company proposal standards.
  6. Approve and publish, or reject with a clear reason.
  7. Notify the requester if additional changes are needed.

Content Library Governance

Admin responsibilities:

  • Approve high-quality reusable content.
  • Reject or return incomplete content.
  • Retire outdated content.
  • Maintain tags and freshness indicators.
  • Ensure regulated claims are reviewed.

Recommended content review questions:

  • Is the content accurate?
  • Is it approved for reuse?
  • Does it contain customer-sensitive data?
  • Does it require CUI or export-control marking?
  • Does it have a clear owner?
  • Is it current?

Data Retention, Archiving, and Records Management

Pursuite data may include proposal drafts, solicitation files, pricing information, customer data, CUI, audit events, AI outputs, and exports.

Recommended Retention Controls

  • Archive proposals after closeout.
  • Retain audit logs according to compliance policy.
  • Retain CUI classification history.
  • Remove unnecessary user access instead of deleting business records.
  • Export required records before contract closeout if the customer or contract requires it.
  • Coordinate purge requests with legal, compliance, and IT.

Step-by-Step: Archive a Proposal

  1. Confirm the proposal is complete or no longer active.
  2. Confirm required exports have been saved.
  3. Confirm final status and opportunity outcome are recorded.
  4. Remove unnecessary workspace users.
  5. Archive the proposal.
  6. Record archive date if required.

Troubleshooting and Escalation

Users Cannot Sign In

Check:

  1. Account is active.
  2. Email address is correct.
  3. SSO is healthy.
  4. MFA enrollment is valid.
  5. User belongs to the correct tenant.
  6. Identity provider group mapping is correct.

Escalate to IT/security if SSO or MFA is failing for multiple users.

User Cannot Access a Proposal

Check:

  1. User has an active account.
  2. User belongs to the proposal workspace.
  3. Workspace role is correct.
  4. System role is sufficient.
  5. Proposal is not archived.
  6. The user is opening the correct tenant.

User Cannot Edit a Section

Check:

  1. Section is assigned to the user or role permits editing.
  2. Section is not locked.
  3. Proposal status permits edits.
  4. User has correct workspace role.
  5. User is not in read-only mode.

RFP Upload or Processing Fails

Check:

  1. File type is PDF or DOCX.
  2. File is under 50 MB.
  3. User is Proposal Manager or Admin.
  4. AI provider is configured.
  5. CUI external-processing gate is not blocking the document.
  6. Worker and queue services are healthy.

RFP Confirmation Is Blocked

Check:

  1. Document status is completed.
  2. User has confirmation permission.
  3. Low-confidence requirements are edited or marked reviewed.
  4. No processing run is currently active.

CUI Processing Is Blocked

Check:

  1. RFP document classification.
  2. Organization external CUI processing approval.
  3. AI provider approval for CUI.
  4. User clearance and access settings.
  5. Audit event details.

AI Features Are Unavailable

Check:

  1. Feature entitlement.
  2. AI provider configuration.
  3. Provider connectivity test.
  4. Usage or budget limits.
  5. Required prerequisite gates.
  6. CUI policy.

Integration Sync Fails

Check:

  1. Integration credentials.
  2. OAuth scope or consent.
  3. Remote service status.
  4. Firewall or air-gap restrictions.
  5. Sync schedule.
  6. Recent sync logs.

Appendix B: Evidence and Audit Export Handling

When exporting evidence:

  1. Confirm the requester and purpose.
  2. Limit the date range.
  3. Filter to relevant users, actions, or objects when possible.
  4. Generate the export.
  5. Store the export in the approved repository.
  6. Restrict access if the export includes CUI, proposal-sensitive, or customer-sensitive information.
  7. Record export date, scope, and owner.
  8. Do not modify raw evidence files.

Recommended evidence naming pattern:

Pursuite_<EvidenceType>_<YYYY-MM-DD>_<Scope>.<extension>

Examples:

  • Pursuite_AuditEvents_2026-06-21_Q2AccessReview.csv
  • Pursuite_CUIAccess_2026-06-21_RFPDocuments.csv
  • Pursuite_UserRoles_2026-06-21_QuarterlyReview.csv
