Task 1: Upload And Process An RFP

End-User Track

Precondition: The user is an Admin or Proposal Manager, the proposal workspace exists, and the file is PDF or DOCX up to 50 MB.

  1. [verified] Open the proposal workspace.
  2. [verified] Open RFP Shredder.
  3. [verified] Select the upload area labeled Drop your RFP here or browse.
  4. [verified] Choose a supported PDF or DOCX file up to 50 MB.
  5. [verified] Wait for the upload to complete.
  6. [verified] Monitor the status until the document is Completed or Failed.
  7. [verified] If the status is Failed, read the displayed processing error before retrying.

Expected result: The document reaches Completed and extracted requirements are visible.

How to confirm success: The requirement count, mandatory count, optional count, and extracted requirement rows appear in the panel.

Admin Track

Precondition: A user reports upload or processing trouble, or an Admin is validating readiness.

  1. [verified] Confirm the user is Admin or Proposal Manager if they need upload rights.
  2. [verified] Confirm the file type is PDF or DOCX.
  3. [verified] Confirm the file is 50 MB or smaller.
  4. [verified] Confirm AI Provider configuration is saved and passes the connectivity test.
  5. [verified] Check worker, queue, API, storage, PostgreSQL, and Redis health through Pursuite Manager logs when processing does not complete.
  6. [verified] If the document is CUI-classified, verify approved external CUI processing before analysis proceeds.

Expected result: The Admin identifies whether the blocker is role, file, AI provider, worker, storage, or CUI policy.

How to confirm success: A corrected upload reaches Completed, or the Admin documents the blocking dependency.

Task 2: Review And Correct Requirements

End-User Track

Precondition: The RFP document status is Completed.

  1. [verified] Review the extracted requirement count.
  2. [verified] Review the mandatory and optional counts.
  3. [verified] Review each extracted requirement row.
  4. [verified] Compare rows marked Review required against the source solicitation.
  5. [verified] Select the edit control for incorrect rows.
  6. [verified] Update requirement text, category, and Mandatory state.
  7. [verified] Select Save.
  8. [verified] Select Add requirement manually for missed requirements.
  9. [verified] Enter requirement text, category, and Mandatory state.
  10. [verified] Select Add.
  11. [verified] Delete false-positive or duplicate rows only after confirming they should be excluded from the active requirement list.
  12. [verified] Select Reviewed for low-confidence rows that are accurate.

Expected result: The requirement list is complete, accurate, and no required review rows remain unresolved.

How to confirm success: The list has no unresolved Review required rows, and all known solicitation requirements are represented.

Admin Track

Precondition: A Proposal Manager asks for help with review controls, low-confidence gates, or row history.

  1. [verified] Confirm the document status is Completed.
  2. [verified] Confirm the user is Admin or Proposal Manager for write actions.
  3. [verified] Explain that low-confidence requirements below the verified threshold must be edited or marked reviewed before confirmation.
  4. [verified] Explain that deleted requirements are soft-deleted rather than hard-deleted.
  5. [verified] Confirm that re-processing uses replace-on-success behavior.

Expected result: The proposal team can finish review without bypassing audit-sensitive behavior.

How to confirm success: Confirmation is no longer blocked by low-confidence requirements.
