LoRA Fine-Tuning Manual Validation Kit

This kit validates the LoRA fine-tuning control plane end to end: consent, preflight, dataset preview, job creation, training manifests, evaluation gates, model registry, promotion, rollback, and audit evidence.

It is designed for manual injection into an existing test tenant. Use an existing admin user so you can validate the UI with the same org that receives the data.

Files

  • 01_seed_lora_training_dataset.sql - opportunity, proposal, sections, section versions, and ai_audit_log rows for dataset preview/export.
  • 02_seed_completed_lora_job_fixture.sql - completed job, manifest, registry entry, passing eval run, and lifecycle events.
  • create_job_payload.json - API payload for a real queued LoRA job.
  • eval_gate_pass_payload.json - evaluation gate config that should allow the completed fixture to promote.
  • eval_gate_fail_payload.json - intentionally strict gate used to verify promotion blocking.

Prerequisites

  1. Apply migrations through 076_fine_tuning_registry_eval_manifest.sql.
  2. Run API and web locally or in the target environment.
  3. Have an admin user you can log in as.
  4. Know that admin user's org_id and user_id.
  5. Set AI_CONFIG_ENCRYPTION_KEY in the API environment; AI provider config encryption requires it.

Find a usable admin:

SELECT id AS user_id, org_id, email, name, system_roles
FROM users
WHERE active = true
ORDER BY created_at DESC
LIMIT 20;

Injection

Run these from the repo root with psql, replacing the two UUIDs:

\set org_id  'YOUR_EXISTING_ORG_UUID'
\set user_id 'YOUR_EXISTING_ADMIN_USER_UUID'
\i docs/testing/lora-fine-tuning-validation/01_seed_lora_training_dataset.sql

To validate promotion/rollback without running the fine-tuner:

\set org_id  'YOUR_EXISTING_ORG_UUID'
\set user_id 'YOUR_EXISTING_ADMIN_USER_UUID'
\i docs/testing/lora-fine-tuning-validation/02_seed_completed_lora_job_fixture.sql

Do not inject 02 into production-like data unless you explicitly want a synthetic completed model registry entry.

Step-By-Step Validation

1. Enable Training Consent

In Admin Console -> AI Provider:

  1. Check Allow org data to be used for local LoRA fine-tuning.
  2. Save configuration.

Expected:

  • Save succeeds.
  • GET /api/v1/admin/ai/config returns "trainingDataOptIn": true.
  • Admin Console -> AI Fine-Tuning -> Preflight shows Training consent as pass.

Evidence query:

SELECT provider, configured_at, updated_at
FROM org_ai_configs
WHERE org_id = 'YOUR_EXISTING_ORG_UUID';

The encrypted value is intentionally not human-readable.

2. Validate Preflight

Open Admin Console -> AI Fine-Tuning.

Expected:

  • Training consent should pass after Step 1.
  • Shared work directory should pass if FINE_TUNING_WORK_DIR exists from the API container/process.
  • Trainable model path may warn if FINE_TUNING_MODEL_PATH is unset. That is acceptable for control-plane validation.
  • Fine-tuner sidecar passes only if the fine-tuner service is running.

API equivalent:

curl -s -H "Authorization: Bearer $TOKEN" \
  http://localhost:3001/api/v1/fine-tuning/preflight

3. Validate Dataset Preview

Admin Console -> AI Fine-Tuning -> Dataset preview -> Preview.

Expected:

  • Exported is at least 12.
  • Eligible is at least 10.
  • Preview rows show hashes, section labels, character counts, and realistic GovCon text snippets.

API equivalent:

curl -s -H "Authorization: Bearer $TOKEN" \
  "http://localhost:3001/api/v1/fine-tuning/dataset-preview?maxSamples=12&previewLimit=5"

Evidence query:

SELECT action, section_type, COUNT(*)
FROM ai_audit_log
WHERE org_id = 'YOUR_EXISTING_ORG_UUID'
  AND session_id = 'manual-lora-validation'
GROUP BY action, section_type
ORDER BY action, section_type;

4. Validate Evaluation Gate Configuration

Admin Console -> AI Fine-Tuning -> Evaluation gate.

Set:

  • Minimum samples: 10
  • Minimum win rate: 0.5
  • Max latency regression: 0.1
  • Require golden set: unchecked
  • Golden set JSON: contents of eval_gate_pass_payload.json's goldenSet

Save Gate.

API equivalent:

curl -s -X PUT \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  --data @docs/testing/lora-fine-tuning-validation/eval_gate_pass_payload.json \
  http://localhost:3001/api/v1/fine-tuning/evaluation-config

Expected:

  • Save succeeds.
  • Audit log contains fine_tuning.eval_config_updated.

Evidence query:

SELECT name, thresholds, jsonb_array_length(golden_set) AS golden_prompts
FROM fine_tuning_eval_configs
WHERE org_id = 'YOUR_EXISTING_ORG_UUID'
ORDER BY updated_at DESC
LIMIT 3;

5. Validate Real Job Creation

Use this only when the fine-tuner service is running or you intentionally want to test failure behavior.

Admin Console -> AI Fine-Tuning:

  1. Base model: llama3.1:8b
  2. Max samples: 12
  3. Queue Fine-Tuning Job.

API equivalent:

curl -s -X POST \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  --data @docs/testing/lora-fine-tuning-validation/create_job_payload.json \
  http://localhost:3001/api/v1/fine-tuning/jobs

Expected if fine-tuner is running:

  • Job appears as pending, then exporting, training, converting, validating, completed.
  • Manifest row is created after export.
  • Registry row is created after model registration.
  • Eval run is created after validation.

Expected if fine-tuner is not running:

  • Preflight sidecar check fails.
  • If queued anyway through API, job should eventually fail with a clear fine-tuner connection error.

6. Validate Manifest, Registry, And Eval Without Real Training

Run 02_seed_completed_lora_job_fixture.sql, then refresh Admin Console -> AI Fine-Tuning.

Expected:

  • Job history shows pursuite-lora-validation-manual:ft-pass as a completed job.
  • Model registry shows pursuite-lora-validation-manual:ft-pass with status registered.
  • The job has a passing eval run.

API checks:

curl -s -H "Authorization: Bearer $TOKEN" \
  http://localhost:3001/api/v1/fine-tuning/models/registry

curl -s -H "Authorization: Bearer $TOKEN" \
  http://localhost:3001/api/v1/fine-tuning/jobs/20000000-0000-0000-0000-000000000001/manifest

curl -s -H "Authorization: Bearer $TOKEN" \
  http://localhost:3001/api/v1/fine-tuning/jobs/20000000-0000-0000-0000-000000000001/evaluations

Evidence query:

SELECT r.model_tag, r.status, r.is_active, e.status AS eval_status, e.failure_reasons
FROM fine_tuning_model_registry r
LEFT JOIN fine_tuning_eval_runs e ON e.id = r.latest_eval_run_id
WHERE r.org_id = 'YOUR_EXISTING_ORG_UUID'
ORDER BY r.created_at DESC;

7. Validate Promotion

Click Promote on the completed fixture job.

Expected:

  • Request succeeds.
  • Registry entry becomes is_active = true, status = promoted.
  • AI provider config switches to provider = custom and encrypted config contains the promoted model.
  • Audit log contains fine_tuning.model_promoted.
  • Model lifecycle events include promoted.

API equivalent:

curl -s -X POST \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  --data '{}' \
  http://localhost:3001/api/v1/fine-tuning/jobs/20000000-0000-0000-0000-000000000001/promote

Evidence query:

SELECT model_tag, status, is_active, promoted_at
FROM fine_tuning_model_registry
WHERE org_id = 'YOUR_EXISTING_ORG_UUID'
ORDER BY updated_at DESC;

SELECT action, target_type, target_id, created_at
FROM audit_events
WHERE org_id = 'YOUR_EXISTING_ORG_UUID'
  AND action IN ('fine_tuning.model_promoted', 'fine_tuning.model_rolled_back', 'fine_tuning.eval_config_updated')
ORDER BY created_at DESC;

8. Validate Rollback

Click Roll Back on the completed fixture job.

Expected:

  • Request succeeds.
  • Previously active registry row becomes is_active = false, status = retired.
  • AI provider custom model is set back to the job base model, llama3.1:8b.
  • Audit log contains fine_tuning.model_rolled_back.
  • Model lifecycle events include rolled_back.

API equivalent:

curl -s -X POST \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  --data '{}' \
  http://localhost:3001/api/v1/fine-tuning/jobs/20000000-0000-0000-0000-000000000001/rollback

9. Validate Promotion Blocking

Use the intentionally strict gate:

curl -s -X PUT \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  --data @docs/testing/lora-fine-tuning-validation/eval_gate_fail_payload.json \
  http://localhost:3001/api/v1/fine-tuning/evaluation-config

Then create or inject a new eval run with status = failed, or run a new real job under the strict thresholds.

Expected:

  • Promotion returns HTTP 409.
  • Response message says the latest evaluation gate did not pass.
  • Active model is not changed.

Manual failed eval insert for the fixture:

INSERT INTO fine_tuning_eval_runs (
  id, org_id, job_id, model_registry_id, eval_config_id, status,
  scores, thresholds, golden_set_summary, failure_reasons
) VALUES (
  '20000000-0000-0000-0000-000000000099',
  'YOUR_EXISTING_ORG_UUID',
  '20000000-0000-0000-0000-000000000001',
  '20000000-0000-0000-0000-000000000004',
  '20000000-0000-0000-0000-000000000003',
  'failed',
  '{"samplesUsed":12,"winRate":0.72,"latencyRegressionPct":-0.0667}'::jsonb,
  '{"minSamples":50,"minWinRateFt":0.95,"maxLatencyRegressionPct":-0.25,"requireGoldenSet":true}'::jsonb,
  '{"promptCount":0,"executionStatus":"not_configured"}'::jsonb,
  '["samples_used 12 is below minimum 50","golden set is required but empty"]'::jsonb
) ON CONFLICT (id) DO NOTHING;

UPDATE fine_tuning_model_registry
SET latest_eval_run_id = '20000000-0000-0000-0000-000000000099'
WHERE id = '20000000-0000-0000-0000-000000000004';

Pass/Fail Checklist

Feature Pass condition
Training consent AI Provider config returns trainingDataOptIn: true; preflight consent check passes.
Preflight Consent and work directory checks pass; sidecar accurately reports pass/fail based on runtime.
Dataset preview Preview returns eligible samples and safe metadata/snippets from seeded proposal content.
Job creation API accepts valid LoRA hyperparameters and enforces one active job per org.
Export config train_config.json uses requested hyperparameters and configured model path behavior.
Manifest fine_tuning_training_manifests has dataset checksum, paths, samples, hyperparameters, stats.
Registry fine_tuning_model_registry records completed model and active/retired state.
Eval config Thresholds and golden-set JSON save and reload through API/UI.
Eval runs fine_tuning_eval_runs stores scores, thresholds, golden-set summary, failure reasons.
Promotion Passing eval promotes model, updates registry, updates AI config, emits audit/event rows.
Promotion block Failed latest eval returns HTTP 409 and does not change active model.
Rollback Base model restored in AI config; registry active model retired; audit/event rows created.

Cleanup

BEGIN;
DELETE FROM fine_tuning_model_events WHERE org_id = 'YOUR_EXISTING_ORG_UUID' AND job_id = '20000000-0000-0000-0000-000000000001';
DELETE FROM fine_tuning_eval_runs WHERE org_id = 'YOUR_EXISTING_ORG_UUID' AND job_id = '20000000-0000-0000-0000-000000000001';
DELETE FROM fine_tuning_model_registry WHERE org_id = 'YOUR_EXISTING_ORG_UUID' AND job_id = '20000000-0000-0000-0000-000000000001';
DELETE FROM fine_tuning_training_manifests WHERE org_id = 'YOUR_EXISTING_ORG_UUID' AND job_id = '20000000-0000-0000-0000-000000000001';
DELETE FROM fine_tuning_jobs WHERE org_id = 'YOUR_EXISTING_ORG_UUID' AND id = '20000000-0000-0000-0000-000000000001';
DELETE FROM ai_audit_log WHERE org_id = 'YOUR_EXISTING_ORG_UUID' AND session_id = 'manual-lora-validation';
DELETE FROM section_versions WHERE org_id = 'YOUR_EXISTING_ORG_UUID' AND id BETWEEN '10000000-0000-0000-0000-000000000201' AND '10000000-0000-0000-0000-000000000209';
DELETE FROM sections WHERE org_id = 'YOUR_EXISTING_ORG_UUID' AND volume_id = '10000000-0000-0000-0000-000000000003';
DELETE FROM volumes WHERE org_id = 'YOUR_EXISTING_ORG_UUID' AND id = '10000000-0000-0000-0000-000000000003';
DELETE FROM proposals WHERE org_id = 'YOUR_EXISTING_ORG_UUID' AND id = '10000000-0000-0000-0000-000000000002';
DELETE FROM opportunities WHERE org_id = 'YOUR_EXISTING_ORG_UUID' AND id = '10000000-0000-0000-0000-000000000001';
COMMIT;

ai_audit_log may be protected by immutability rules in some environments. If deletion is blocked, leave the seeded audit rows in the test tenant or restore from a test snapshot.
