Version: 1.0
Scenario: DHS ICE Zero Trust Help Desk Modernization
Seed location: app/test-data/manual-validation

1. Purpose

This guide gives a manual tester a realistic GovCon dataset and a step-by-step workflow to confirm that Pursuite is working as defined across the major product surfaces:

  • Login, tenant isolation, roles, and dashboard visibility
  • CRM accounts, contacts, interactions, pipeline, and opportunity records
  • Proposal workspace, volumes, sections, comments, versions, and win themes
  • RFP Shredder output, document chunk corpus, compliance matrix, and traceability
  • Content Library, pricing/quotes, pWin, competitive intel, and capture intel
  • AI/RAG Q&A, drafting context, abstention, citation grounding, and tenant-leakage checks
  • Color review, autonomous pipeline state, teaming workspace, templates, workflows, and restricted pricing volume access

The dataset is intentionally small but connected. One scenario threads through the system so a tester can verify whether each feature is pulling the correct tenant-scoped evidence.

3. Prerequisites

  1. Use a local or disposable Pursuite database. Do not inject this data into a production tenant.
  2. Ensure all normal app migrations have run.
  3. Confirm Postgres is pgvector-backed. The standard standalone/production compose file should use pgvector/pgvector:pg16.
  4. Confirm Redis is running if you want to log in through local auth.
  5. Have either psql available on the host or use docker compose exec postgres psql.
  6. Optional but recommended: configure an AI provider in Admin > AI Settings for positive AI generation checks. Without an AI provider, seeded UI/data checks and Q&A abstention can still be validated, but grounded positive answers will return 503 AI Not Configured.

4. Inject The Dataset

From the repository root (H:\Projects\Pursuite\app):

psql "$env:DATABASE_URL" -f "test-data/manual-validation/manual_validation_cleanup.sql"
psql "$env:DATABASE_URL" -f "test-data/manual-validation/manual_validation_seed.sql"
psql "$env:DATABASE_URL" -f "test-data/manual-validation/manual_validation_smoke_checks.sql"

If you are using the standalone compose stack with host Postgres on port 5432, set a database URL like this:

$env:DATABASE_URL = "postgres://pursuite:<POSTGRES_PASSWORD>@localhost:5432/pursuite"

If you are using the local-test overlay from this repo, host Postgres is usually on port 5433:

$env:DATABASE_URL = "postgres://pursuite:<POSTGRES_PASSWORD>@localhost:5433/pursuite"

If you prefer running inside the container:

docker compose -f docker-compose.production.yml --env-file .env.production exec -T postgres psql -U pursuite -d pursuite -f /path/in/container/manual_validation_seed.sql

After the SQL load, set a local admin password. Local auth is admin-only and stores its password hash in Redis, so SQL alone cannot create the login password.

Set-Location H:\Projects\Pursuite\app\apps\api
npm.cmd run reset-local-pw -- --email validation.admin@pursuite.local --org manual-validation --password "PursuiteManual!2026"

Login values:

Field Value
Org slug manual-validation
Admin email validation.admin@pursuite.local
Password PursuiteManual!2026 unless you chose a different value

7. AI And RAG Validation

The strongest RAG checks are listed in manual_validation_cases.json. Use these in the UI chat/Q&A surface or direct API calls.

7.1 Get A Token For API Checks

$base = "http://localhost"
$loginBody = @{
  email = "validation.admin@pursuite.local"
  password = "PursuiteManual!2026"
  orgSlug = "manual-validation"
} | ConvertTo-Json

$login = Invoke-RestMethod -Method Post -Uri "$base/api/v1/auth/local" -ContentType "application/json" -Body $loginBody
$token = $login.data.accessToken
$headers = @{ Authorization = "Bearer $token" }

7.2 Provider Check

Invoke-RestMethod "$base/api/v1/ai/provider" -Headers $headers

Expected:

  • If configured: provider name and tool capability.
  • If not configured: provider = not_configured. Positive grounded answer checks will return 503; abstention can still be checked.

7.3 Q&A Abstention Check

This test should pass even without an AI provider because the route hard-abstains before provider selection when sources are absent.

$body = @{
  question = "What is the guaranteed award fee for this RFP?"
  proposalId = "40000000-0000-4000-8000-000000000001"
} | ConvertTo-Json

Invoke-RestMethod -Method Post -Uri "$base/api/v1/ai/ask" -Headers $headers -ContentType "application/json" -Body $body

Expected:

  • abstained = true
  • Answer says the information was not found in source.
  • No citations.

12. Notes And Known Boundaries

  • This seed does not create real files in storage. It inserts file metadata and source text fixtures. Upload/download tests should still be performed with real uploads.
  • The SQL does not populate vector embeddings. Sparse and exact-match retrieval paths are intentionally covered by the text and identifier data. To test dense vector recall, configure embeddings and run the app's reindex/embedding flows after seeding.
  • Positive AI answer/drafting checks require a configured AI provider. Without one, use the UI/data checks and abstention test.
  • Local auth is admin-only. Non-admin seeded users are present for role and workspace data, but they cannot use /auth/local unless SSO or another login path is configured.
  • The canary tenant is intentionally similar to the primary tenant. It exists to catch tenant leakage.
