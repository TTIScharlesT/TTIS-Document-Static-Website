TSG License Manager GCP Staging Deployment Steps

This runbook deploys TSG License Manager to GCP staging using Cloud Run, Cloud SQL for PostgreSQL, Secret Manager, and Artifact Registry.

Staging values

GCP_PROJECT_ID=ttis-ai
GCP_REGION=us-central1
ARTIFACT_REPOSITORY=tsg-platform
ARTIFACT_IMAGE=tsg-license-manager
CLOUD_RUN_SERVICE=tsg-license-manager-staging
CLOUD_SQL_INSTANCE=tsg-license-manager-staging
CLOUD_SQL_DATABASE=tsg_license_manager
CLOUD_SQL_USER=tsg_app
STAGING_BASE_URL=https://license-staging.tti-solutions.com

1. Authenticate GCP tools

gcloud auth login
gcloud config set project ttis-ai
gcloud auth configure-docker us-central1-docker.pkg.dev
gcloud config get-value project
gcloud auth list

2. Enable required GCP APIs

gcloud services enable `
  run.googleapis.com `
  sqladmin.googleapis.com `
  secretmanager.googleapis.com `
  artifactregistry.googleapis.com `
  cloudbuild.googleapis.com

3. Create Artifact Registry repository

gcloud artifacts repositories create tsg-platform `
  --repository-format=docker `
  --location=us-central1 `
  --description="TSG License Manager container images"

If the repository already exists, record that and continue.

4. Build and push the container image

cd G:\Tools\tsg-platform-v0.2.0-dev-sprint-n

docker build --tag tsg-platform:0.2.0-rc.1 .

docker tag tsg-platform:0.2.0-rc.1 `
  us-central1-docker.pkg.dev/ttis-ai/tsg-platform/tsg-license-manager:0.2.0-rc.1

docker push us-central1-docker.pkg.dev/ttis-ai/tsg-platform/tsg-license-manager:0.2.0-rc.1

Record the pushed image digest in the release evidence.

5. Create Cloud SQL PostgreSQL

gcloud sql instances create tsg-license-manager-staging `
  --database-version=POSTGRES_17 `
  --tier=db-f1-micro `
  --region=us-central1 `
  --storage-type=SSD `
  --storage-size=10GB `
  --availability-type=zonal
gcloud sql databases create tsg_license_manager `
  --instance=tsg-license-manager-staging
gcloud sql users create tsg_app `
  --instance=tsg-license-manager-staging `
  --password="REPLACE_WITH_STRONG_PASSWORD"

Use a strong generated password. Do not commit it.

6. Create Secret Manager secrets

Required secrets:

DATABASE_URL
TSG_AUTH_SECRET
TSG_LICENSE_SIGNING_SECRET
TSG_CLIENT_API_SECRET
STRIPE_WEBHOOK_SECRET
SMTP_PASSWORD

The Cloud Run / Cloud SQL Unix-socket DATABASE_URL should look like:

postgresql://tsg_app:YOUR_PASSWORD@/tsg_license_manager?host=/cloudsql/ttis-ai:us-central1:tsg-license-manager-staging
echo "postgresql://tsg_app:YOUR_PASSWORD@/tsg_license_manager?host=/cloudsql/ttis-ai:us-central1:tsg-license-manager-staging" | gcloud secrets create tsg-license-manager-staging-database-url --data-file=-

echo "YOUR_AUTH_SECRET" | gcloud secrets create tsg-license-manager-staging-auth-secret --data-file=-

echo "YOUR_LICENSE_SIGNING_SECRET" | gcloud secrets create tsg-license-manager-staging-license-signing-secret --data-file=-

echo "YOUR_CLIENT_API_SECRET" | gcloud secrets create tsg-license-manager-staging-client-api-secret --data-file=-

echo "YOUR_STRIPE_WEBHOOK_SECRET" | gcloud secrets create tsg-license-manager-staging-stripe-webhook-secret --data-file=-

echo "YOUR_SMTP_PASSWORD" | gcloud secrets create tsg-license-manager-staging-smtp-password --data-file=-

PowerShell note: if echo introduces formatting issues, write each secret to a temporary local file, create the secret with --data-file, then securely delete the temporary file.

7. Deploy Cloud Run

gcloud run deploy tsg-license-manager-staging `
  --image us-central1-docker.pkg.dev/ttis-ai/tsg-platform/tsg-license-manager:0.2.0-rc.1 `
  --region us-central1 `
  --platform managed `
  --add-cloudsql-instances ttis-ai:us-central1:tsg-license-manager-staging `
  --set-env-vars TSG_ENV=staging,TSG_DB_BACKEND=postgres,TSG_SECURE_COOKIES=true,TSG_PUBLIC_BASE_URL=https://license-staging.tti-solutions.com,TSG_ALLOWED_HOSTS=license-staging.tti-solutions.com,docguard-ai-staging.tti-solutions.com,pursuite-staging.tti-solutions.com `
  --set-secrets DATABASE_URL=tsg-license-manager-staging-database-url:latest,TSG_AUTH_SECRET=tsg-license-manager-staging-auth-secret:latest,TSG_LICENSE_SIGNING_SECRET=tsg-license-manager-staging-license-signing-secret:latest,TSG_CLIENT_API_SECRET=tsg-license-manager-staging-client-api-secret:latest,STRIPE_WEBHOOK_SECRET=tsg-license-manager-staging-stripe-webhook-secret:latest,SMTP_PASSWORD=tsg-license-manager-staging-smtp-password:latest `
  --allow-unauthenticated

Ingress can be restricted later if the existing Virtual Business Solution owns routing.

8. Map staging domain

gcloud run domain-mappings create `
  --service tsg-license-manager-staging `
  --domain license-staging.tti-solutions.com `
  --region us-central1

Add the DNS records returned by Google at the DNS provider.

Repeat later if product portals route directly to this service:

docguard-ai-staging.tti-solutions.com
pursuite-staging.tti-solutions.com

9. Verify deployment

curl https://license-staging.tti-solutions.com/health
curl https://license-staging.tti-solutions.com/ready

Expected /ready result:

database=ok
status=ready
gcloud run services logs read tsg-license-manager-staging `
  --region us-central1 `
  --limit 100

10. Run staging acceptance smoke checks

  • /health returns OK.
  • /ready returns ready with database OK.
  • Admin login does not expose development credentials.
  • DocGuard AI portal route resolves.
  • Pursuite portal route resolves.
  • DocGuard AI product appears as docguard_ai.
  • No retired development codename appears in customer-facing evidence.
  • Client API rejects missing credentials.
  • Client API accepts valid DocGuard AI credentials.
  • Stripe signed test webhook passes.
  • Stripe tampered webhook fails.
  • SMTP password setup email sends.

Closure evidence to record

  • GCP project ID.
  • Cloud Run service URL.
  • Cloud Run revision.
  • Image path and digest.
  • Cloud SQL instance/database.
  • Secret Manager secret names.
  • /health result.
  • /ready result.
  • Smoke-test timestamp.
  • Operator name.
  • Any DNS/domain mapping records.
  • Links or screenshots for successful acceptance checks.
