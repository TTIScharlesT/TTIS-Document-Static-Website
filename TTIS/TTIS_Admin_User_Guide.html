1. Purpose And Scope

This guide explains how to administer the TTIS AI Virtual Business System after Dashboard API v3. It covers day-to-day administration, security controls, user lifecycle, webhook trust, workflow gates, deployment operations, and audit evidence.

The guide does not replace source-controlled implementation references. For API-level contracts, use TTIS_AI_API_Reference_v3.md. For deep production hosting details, use docs/PRODUCTION_HOSTING.md.

2. Admin Responsibilities

Administrators are responsible for keeping the system usable, secure, auditable, and current.

Core duties:

  • Provision and deactivate users.
  • Assign correct ABAC roles, divisions, auditor posture, and any explicit grants.
  • Verify MFA enrollment and help users recover from lost devices.
  • Maintain environment secrets and encryption keys.
  • Monitor workflow gates, failed jobs, webhook drops, and service health.
  • Maintain the Knowledge corpus and ensure sensitive data remains ABAC-scoped.
  • Confirm that Redis, dashboard, webhook, worker, and ChromaDB are running in the expected deployment shape.
  • Preserve the audit trail and never disable chain verification.

3. System Access Model

TTIS uses per-user identity and ABAC capability authorization. There is no shared dashboard bearer secret in the v3 model.

3.1 Roles

Role Typical user Administrative meaning
ceo Owner / executive administrator Superuser role; can administer users, agents, policy, gates, workflows, finance, payroll, Knowledge, and Ask-LYNNE.
field_ops_mgr Field operations manager Division-scoped management of projects, schedule, safety, timecards, onboarding, documents, and OPS EAA.
business_dev Business development user Pipeline, workflow, document, telemetry, and business-development operations.
finance Finance/payroll authority Finance, payroll, invoice, audit export, travel booking/approval, and finance-level approvals.
field_crew Technician / crew member Own timecards, daily logs, incidents, travel requests, schedule, LMS, onboarding, and field views.

Auditor posture is an additional attribute. An auditor can read permitted resources but is denied mutating actions.

3.2 Capabilities

Authorization is evaluated as:

principal + resource + action + context -> allow or deny

Examples:

  • finance can approve finance-level timecards.
  • field_ops_mgr can manage division-scoped schedules.
  • field_crew can submit their own timecards and daily logs.
  • ceo can access Ask-LYNNE and global administration.

The server is the security boundary. Do not rely on UI hiding alone.

4. First-Time Admin Setup

4.1 Confirm Environment Configuration

  1. Open config/.env.
  2. Confirm these minimum values are set:
ANTHROPIC_API_KEY=
DASHBOARD_JWT_SECRET=
DASHBOARD_PWD_SALT=
CORS_ALLOW_LIST=
REDIS_URL=
SQLITE_DIR=
CHROMA_DIR=
DOCUMENTS_DIR=
PURSUITE_WEBHOOK_SECRET=
TELEGRAM_WEBHOOK_SECRET=
AUTH_KEY_FILE=/app/secrets/auth.key
  1. Confirm DASHBOARD_JWT_SECRET is at least 32 characters and not a placeholder.
  2. Confirm DASHBOARD_PWD_SALT is at least 16 characters.
  3. Confirm CORS_ALLOW_LIST contains the real dashboard origin.
  4. Confirm AUTH_KEY_FILE points to a protected secrets volume, or AUTH_ENC_KEY is set as an environment fallback.

4.2 Install Dependencies

From the repository root:

pip install -r requirements.txt

The password hashing stack requires argon2-cffi.

4.3 Start Local Services

Development mode:

python main.py
uvicorn dashboard.main:app --host 127.0.0.1 --port 8000
uvicorn tools.pursuite_webhook:webhook_app --host 127.0.0.1 --port 8001

Production-shaped queue mode:

docker compose up -d

Expected local services:

Service Local endpoint Notes
Dashboard API http://127.0.0.1:8000 Caddy-routed in production.
Webhook receiver http://127.0.0.1:8001 Caddy-routed in production.
Redis 127.0.0.1:6379 Localhost only; never public.
ChromaDB 127.0.0.1:8002 Localhost only.

4.4 Verify Health

  1. Open http://127.0.0.1:8000/health.
  2. Open http://127.0.0.1:8001/webhooks/health.
  3. Confirm Redis responds:
docker exec ttis-redis redis-cli ping

Expected response: PONG.

5. User Administration

5.1 User Lifecycle Overview

Stage Admin action Result
Create Add user with role, division, password, and optional grants User can sign in and receives ABAC-scoped capabilities.
Enroll MFA User enrolls TOTP Sensitive actions can be step-up authorized.
Update Change role/division/grants as duties change Effective capabilities update on future requests.
Reset Admin resets password when required New password is stored as Argon2id.
Disable Deactivate account User can no longer authenticate.

5.2 Create A User

Who can perform: ceo or an explicitly authorized user-admin operator.

Recommended steps:

  1. Open the dashboard administration area.
  2. Select Users.
  3. Choose Create User.
  4. Enter: - user ID - username - temporary password - ABAC role - division, when applicable - auditor flag, when applicable - optional wizard grants, if the user needs wizard access
  5. Save.
  6. Provide the temporary password through an approved secure channel.
  7. Instruct the user to sign in, change the password, and enroll MFA.

Expected result: the new user appears in the user list and can authenticate.

5.3 Choose The Correct Role

Use least privilege.

If the user needs to... Assign
administer the whole platform ceo
manage crews, schedule, timecards, incidents, or onboarding for a division field_ops_mgr
work pipeline, business development, documents, and opportunities business_dev
approve finance, payroll, invoices, audit export, or travel booking finance
submit field work, time, incidents, travel, LMS, and onboarding tasks field_crew

5.4 Reset A Password

  1. Open Admin > Users.
  2. Select the user.
  3. Choose Reset Password.
  4. Enter a temporary password that satisfies the policy.
  5. Save.
  6. Tell the user to sign in and immediately change it.

Expected result: the stored password is Argon2id. Legacy salted SHA-256 values remain accepted only for migration and are upgraded after successful login.

5.5 Disable A User

Disable users immediately when employment ends, duties change, a device is lost, or credentials are suspected compromised.

  1. Open Admin > Users.
  2. Select the user.
  3. Toggle active status to disabled.
  4. Save.
  5. Review audit logs for recent activity.

Expected result: future login attempts fail.

6. MFA And Step-Up Administration

6.1 What Step-Up Protects

Step-up MFA protects sensitive and high-impact actions such as:

  • large financial approvals
  • payroll-sensitive actions
  • policy threshold increases
  • sensitive key generation or rotation
  • governed workflow decisions above threshold

6.2 User Enrollment

  1. User signs in.
  2. User opens security/MFA enrollment.
  3. User scans the TOTP secret with an authenticator app.
  4. User stores backup codes securely.
  5. User verifies a current code.

Expected result: GET /api/v1/auth/me reports totp_enrolled: true.

6.3 Admin Recovery

If a user loses MFA access:

  1. Confirm identity using company-approved verification.
  2. Disable the account temporarily if compromise is suspected.
  3. Reset password if needed.
  4. Clear or refresh MFA enrollment only through the approved admin path.
  5. Require the user to enroll a new authenticator.
  6. Review audit events.

6.4 Encryption Key Requirements

TOTP secrets are encrypted at rest. The app reads key material from:

  1. AUTH_KEY_FILE, default /app/secrets/auth.key
  2. AUTH_ENC_KEY, environment fallback

File precedence is intentional. Do not print, log, email, or paste key material into tickets.

7. Role And Capability Management

7.1 Capability Review Procedure

Perform capability reviews:

  • when a user is created
  • when a user changes duties
  • when an auditor posture is assigned
  • before external audits
  • after security incidents

Steps:

  1. Export or inspect the user list.
  2. Confirm each role matches current job duties.
  3. Confirm division assignments are correct.
  4. Confirm auditor users cannot mutate records.
  5. Confirm any explicit grants are still justified.
  6. Remove stale grants.
  7. Record the review outcome.

7.2 Division Scope

Division-scoped resources include projects, safety, incidents, timecards, travel, schedule, and onboarding. Do not assign a broad role to solve a division mismatch. Correct the division attribute instead.

9. Workflow And Gate Administration

9.1 Gate Review

  1. Open Gates or Overview.
  2. Review gate type, workflow ID, presenter, age, confidence, and details.
  3. Confirm the underlying evidence.
  4. Approve, reject, or request changes where supported.
  5. Complete step-up if prompted.
  6. Confirm the result appears in workflow trace and audit.

9.2 Stalled Workflow Handling

  1. Open the workflow detail or trace.
  2. Identify the current step and last successful transition.
  3. Check whether a human gate is waiting.
  4. Check worker logs if queue mode is enabled.
  5. Confirm Redis queue depth if jobs are not draining.
  6. Use pause/override/terminate controls only when authorized.
  7. Record the decision reason.

9.3 Override Safety

Overrides are sensitive. Use them only when the normal gate path cannot resolve the issue in time.

Before override:

  • verify workflow ID
  • verify business justification
  • verify authorization
  • complete step-up if required
  • capture the reason in the action comments

10. Agent And EAA Administration

10.1 Agent Status Review

  1. Open Agents.
  2. Review health, status, model tier, recent actions, flags, and degraded state.
  3. Investigate repeated failures or tool errors.
  4. Confirm no disabled agent is required for an active workflow.

10.2 EAA Configuration

Editable EAAs include LYNNE and the OPS support/management assistants.

Steps:

  1. Open agent detail.
  2. Choose Manage Config.
  3. Review persona, tools, knowledge sources, and limits.
  4. Make the minimum necessary change.
  5. Save.
  6. Complete step-up if prompted.
  7. Confirm the change applies on the next task.

10.3 Disable Or Force-Terminate

  1. Open the agent detail.
  2. Confirm the reason.
  3. Disable or force-terminate only when the agent is unsafe, looping, or blocking work.
  4. Confirm worker and dashboard both honor the change.
  5. Review audit events.

11. Knowledge Base Administration

11.1 Corpus Ownership

The Knowledge corpus may include policy, regulations, project documents, proposal material, and other sensitive data. Treat it as controlled internal content.

11.2 Add Or Refresh Documents

  1. Place approved documents in the configured documents directory.
  2. Run the ingestion process or use the dashboard refresh action when available.
  3. Confirm the document appears in Knowledge search.
  4. Search for a known phrase from the document.
  5. Confirm citations and section views are correct.

11.3 Quality Checks

For each corpus update:

  • confirm the source is approved
  • confirm classification is appropriate
  • confirm no prohibited PII is exposed
  • confirm relevant roles can find it
  • confirm unauthorized roles cannot see restricted content

12. Finance, Payroll, And Sensitive Actions

12.1 Finance Administration

Finance administrators manage contracts, cash flow, labor postings, invoices, audit export, and finance-level approvals.

Before approving:

  1. Confirm amount, contract, source records, and attachments.
  2. Confirm provenance from timecard or invoice line.
  3. Confirm all required prior approvals exist.
  4. Complete step-up if prompted.
  5. Confirm audit hash is recorded.

12.2 Payroll Administration

Payroll actions are sensitive and role-gated.

  1. Confirm employee records are current.
  2. Confirm approved timecards and labor postings.
  3. Generate or review the payroll run.
  4. Review deductions and tax settings.
  5. Generate ACH/NACHA only when authorized.
  6. Confirm encrypted bank detail handling and key status.
  7. Preserve generated artifacts as required by policy.

12.3 Key Status

Payroll encryption keys and auth-side TOTP encryption keys must live in protected secrets storage. Never return key material from an endpoint, log, screenshot, or ticket.

13. Webhook Administration

13.1 Pursuite Webhook

Pursuite webhook requests require:

  • X-Pursuite-Signature
  • X-Pursuite-Timestamp, required by default
  • optional X-Pursuite-Delivery, used for replay detection

Environment controls:

PURSUITE_REQUIRE_TIMESTAMP=1
PURSUITE_MAX_TIMESTAMP_SKEW_SECONDS=300
PURSUITE_REPLAY_TTL_SECONDS=86400

Validation behavior:

Failure Result
Invalid signature 401
Missing required timestamp 401
Stale timestamp 401
Duplicate delivery 409
Unknown event type Safe ignored response

13.2 Telegram Webhook

Telegram requires:

  • URL token equal to TELEGRAM_WEBHOOK_SECRET
  • X-Telegram-Bot-Api-Secret-Token header equal to the same secret
  • sender chat ID in TG_USER_WHITELIST

Registration:

python scripts/register_telegram_webhook.py --public-url https://webhook.example.com

Expected result: Telegram sends the secret header and the receiver accepts only whitelisted chats.

14. Deployment And Runtime Operations

14.1 Production Shape

Production should expose only Caddy publicly. Dashboard, webhook, Redis, and ChromaDB should be private or localhost-bound.

Component Exposure rule
Caddy Public 80/443
Dashboard Caddy-routed, not directly public
Webhook Caddy-routed, not directly public
Redis Never public
ChromaDB Never public
Secrets volume Mounted, protected, app-user owned

14.2 Docker Hardening Expectations

Dashboard and webhook images run as the non-root ttis user. Compose applies:

cap_drop:
  - ALL
security_opt:
  - no-new-privileges:true

Writable mounted volumes must be owned by the app user.

14.3 Deployment Validation

Run:

docker compose config
docker build -f Dockerfile.dashboard -t ttis-dashboard:hardening-check .
docker build -f Dockerfile.webhook -t ttis-webhook:hardening-check .

Expected result:

  • compose config renders successfully
  • dashboard image builds
  • webhook image builds
  • image user is ttis

15. Monitoring, Audit, And Evidence

15.1 Audit Trail

The audit trail is append-only and chain-verified. Never disable, truncate, or rewrite audit data.

Admin actions that should produce audit evidence:

  • user creation, disablement, password reset
  • gate decision
  • workflow override
  • agent configuration change
  • payroll key generation or rotation
  • finance approval
  • audit export
  • webhook security drop

15.2 Daily Evidence Review

  1. Verify audit chain.
  2. Review failed login patterns.
  3. Review webhook security drops.
  4. Review high-value approvals.
  5. Review disabled or degraded agents.
  6. Review worker errors and queue backlog.

16. Backup, Recovery, And Key Handling

16.1 Backup Scope

Back up:

  • SQLite databases or PostgreSQL database
  • ChromaDB persistence
  • document corpus
  • audit database
  • secrets volumes
  • environment configuration templates without secret values

16.2 Recovery Order

  1. Restore database state.
  2. Restore secrets volume.
  3. Restore document and Chroma stores.
  4. Start Redis.
  5. Start dashboard, webhook, worker, and ChromaDB.
  6. Verify health endpoints.
  7. Verify audit chain.
  8. Run a login test.
  9. Run a Knowledge search test.
  10. Run a safe webhook health test.

16.3 Key Handling Rules

  • Store encryption keys in secrets volumes or managed secret stores.
  • Prefer file-backed keys over environment fallback.
  • Restrict filesystem ownership to the app user.
  • Rotate keys only through approved procedures.
  • Never include key material in screenshots, chat, tickets, or logs.

17. Security Checklist

Before production go-live or after major changes:

  • [ ] DASHBOARD_JWT_SECRET is set, strong, and not shared.
  • [ ] DASHBOARD_PWD_SALT is set and strong.
  • [ ] AUTH_KEY_FILE or AUTH_ENC_KEY is configured.
  • [ ] Users have individual accounts.
  • [ ] MFA enrollment is enforced for sensitive users.
  • [ ] Roles and divisions match job duties.
  • [ ] Auditor users cannot mutate records.
  • [ ] Redis is not public.
  • [ ] Webhook is Caddy-routed, not directly public.
  • [ ] Pursuite timestamp validation is required in production.
  • [ ] Telegram header secret is enforced.
  • [ ] Docker images run as non-root.
  • [ ] Audit chain verification passes.
  • [ ] Backups include databases, corpus, and secrets volumes.
  • [ ] Full regression or curated security suite has been run.

18. Troubleshooting

Symptom Likely cause Admin action
Login returns 503 Auth config missing or weak Check DASHBOARD_JWT_SECRET, salt, user DB bootstrap.
Login returns 401 Wrong credentials or disabled user Reset password or reactivate only after verification.
User cannot see a page ABAC role/division mismatch Check role, division, auditor flag, and grants.
Sensitive action returns step-up error TOTP not enrolled or assertion expired Enroll/verify TOTP and retry.
Pursuite returns 401 Bad HMAC or timestamp Check secret, timestamp header, clock skew.
Pursuite returns 409 Replay detected Confirm whether provider retried the same delivery ID.
Telegram silently drops update Header/path secret mismatch or chat not whitelisted Re-register webhook and verify TG_USER_WHITELIST.
Workflow not progressing Gate pending, Redis unavailable, or worker down Review gate queue, Redis, worker logs.
Knowledge search is empty Corpus not ingested or Chroma unavailable Re-ingest and verify Chroma health.
Docker service cannot write Mounted volume ownership mismatch Fix ownership for the ttis app user.
